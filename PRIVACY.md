# VeryLoving Privacy Notes

VeryLoving is a personal safety companion. The app requests sensitive permissions only for safety features the user chooses to use.

## Data Used By Core Features

- Account data: name, email, phone verification status, and sign-in provider. For Apple/Google, the provider identity token and Apple nonce are sent over HTTPS to the VeryLoving backend for signature/issuer/audience/expiry verification. The app then stores separate access and refresh JWTs in SecureStore, renews before access expiry, and rotates the client-held refresh token rather than using the provider assertion as its app session. Refresh tokens remain stateless/replayable until expiry because persistent family revocation/reuse detection is not yet implemented.
- Location: current location for the safety map, danger zones, quick share, safe-arrival context, and SOS context. The most recent valid live coordinate is retained locally for no more than 24 hours so the app can show a timestamped, explicitly stale fallback. A bounded native Mapbox tile pack around the last live location may remain until it is replaced, sign-out, or local-data deletion. When the safety backend is enabled, a user-confirmed SOS may send a recent coordinate to the VeryLoving backend and persist it with the SOS record in DynamoDB.
- Audio and conversation content: during an online AI companion call, the app captures 48 kHz mono Int16 microphone buffers and sends base64 PCM frames through the authenticated VeryLoving WebSocket gateway to Hume. When the custom language model is enabled, Hume sends conversation text, emotional context, and an opaque custom session ID to the VeryLoving CLM service so it can produce a safety-focused response.
- Bluetooth: nearby NorthStar/VL01 wearable identifiers and connection state.
- Notifications: local or remote safety reminders and emergency updates.
- Conversation history: AI companion messages and Hume chat-group references are saved locally on the device. Typed messages created offline are queued locally until the user reconnects to voice AI.
- Emergency and wearable fallback data: emergency contacts, the time and non-PII outcome of the last SOS attempt, and the last paired NorthStar identifier/reconnect state are stored locally. When the production safety backend is enabled, contacts are also fetched/created/deleted through the account-authenticated API, safety sessions are persisted, and SOS acceptance stores contact IDs plus an optional recent location in DynamoDB. An `accepted` response means stored, and a `dialer_opened` outcome means only that the phone dialer opened; neither confirms guardian delivery, call connection, or emergency dispatch.
- Permission rationale state: the app stores whether each contextual permission explanation has already been shown so it does not repeatedly interrupt the user. The native operating system remains the authority for the actual permission decision.
- Diagnostics and usage: native SDK performance, crash, and product interaction data from bundled platform SDKs such as Expo, React Native, Google Sign-In, and Mapbox.

## User Controls

- Settings > Privacy & data > Export my data creates a JSON export containing the local account profile, settings, account-bound emergency-contact cache, conversation history, permission-rationale state, and other `veryloving.*` resilience records. When the safety backend is enabled, it also fetches the authenticated account's DynamoDB contacts, current safety state, and SOS records. The app writes the file to its cache only long enough to open the native share sheet and uses a `finally` cleanup to delete its temporary copy even when sharing fails. A copy saved or sent through the selected share destination is controlled by that destination.
- Settings > Privacy & data > Delete all local data drains tracked writes, removes app-scoped local storage and SecureStore auth entries, and attempts to purge cached voice files and every app-owned Mapbox offline pack. A native artifact failure cannot prevent credential and PII deletion, but the app warns that cleanup was incomplete. It retains only opaque pack identifiers or an enumeration-needed flag outside the user-data namespace so a later native cleanup attempt can be verified; legacy pack names that encoded coordinates are never copied into this retry record.
- Settings > Offline mode uses bundled companion responses when Hume or the network is unavailable.

When the safety backend is enabled, Delete My Data first sends an authenticated `DELETE /v1/privacy/data` request that removes the account's current DynamoDB records, then clears local stores and credentials. A remote failure stops the flow and is shown to the user rather than falsely claiming completion. This endpoint does not yet revoke the session or create an account-deletion tombstone, so a still-valid token could theoretically recreate data until revocation/expiry. It also does not delete Hume, identity-provider, Mapbox, share-destination, logging, backup, or other vendor copies. Production tombstone/revocation, backup/TTL timing, vendor retention, export, and deletion orchestration must be approved and verified before release.

The server is designed not to log message text, microphone frames, access tokens, raw custom session IDs, contact values, or precise location. CLM diagnostics use request counts and a short one-way hash of the opaque session ID; gateway authentication failures log only an error class. The ingress, AWS SDK, DynamoDB, Hume, and any upstream model require independent log, encryption, retention, access, export, and deletion review. Production behavior must match the public privacy policy and store disclosures.

Emergency-contact cache PII is now stored in an account-bound SecureStore envelope and its legacy plaintext key is removed during migration. Paired-device metadata is account-bound but, along with cached location, conversation data, offline queues, settings, permission state, and other resilience records, still uses plaintext AsyncStorage. Those stores require complete account binding where missing, OS-protected encryption, migration, and account-switch/process-death testing. These remain P1 stop-ship gates in `LAUNCH_CHECKLIST.md`; local fallbacks do not make the current build production-ready.

The DynamoDB safety table contains account-partitioned contact PII and may contain recent SOS location and current safety-state records. It supports an account-scoped export and batch deletion; transient SOS records carry `expiresAt` metadata, but production must enable TTL on that attribute. Encryption, IAM, backups, TTL timing, retention/deletion tombstones, support access, breach response, export completeness, and regional-transfer controls still require approval and verification. The current API persists SOS acceptance but does not implement notification delivery or delivery receipts.

## Permission And Store-Disclosure Review

Location, notifications, microphone, and Bluetooth support active safety, voice, map, and wearable flows. The native configuration also declares camera and photo-library access for user-initiated profile or safety images, but the current client has no active image-capture or image-picker flow. Before store submission, Product and Privacy must either validate an implemented user-initiated flow or remove the unused declarations; store disclosures and reviewer notes must describe the behavior actually shipped.

## App Store Privacy Manifest

The iOS privacy manifest is generated from `app.config.js`. It declares linked account, device, location, audio, and user-content data plus unlinked product-interaction, performance, crash, and diagnostic data for app functionality and analytics. It declares no cross-app tracking. These declarations are a conservative source configuration, not a substitute for reconciling the generated archive, SDK privacy manifests, production vendors, App Store privacy answers, and the public privacy policy before every release.
